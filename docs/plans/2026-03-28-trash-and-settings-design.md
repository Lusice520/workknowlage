# Trash And Settings Modal Design

**Goal**

完成“安全删除”这一期产品迭代，把文档与文件夹删除从硬删除改成回收站语义，同时把设置面板从空间切换下拉中迁移为独立弹窗。

**Confirmed Scope**

本期只做：

- 文档进入回收站
- 文件夹整包进入回收站
- 当前空间独立回收站
- 回收站中的恢复
- 回收站中的彻底删除
- 清空当前空间回收站
- 设置入口改为弹窗

本期明确不做：

- 空间进入回收站
- 快记进入回收站
- 全局跨空间回收站
- 合并恢复
- 版本化历史删除记录

**Current Problem**

- 当前文档、文件夹、空间删除都是直接硬删除，误操作没有恢复余地。
- 左侧 `回收站` 只是一个占位按钮，没有真实行为。
- 现有设置功能嵌在空间切换浮层内部，信息密度已经偏高，再继续扩展会让交互变得拥挤。
- 文档分享、搜索索引、附件清理目前都默认围绕“活跃文档”工作，删除语义变化后需要统一收口。

## Decision

采用“主表软删除 + 当前空间回收站视图”的方案，不引入单独的 trash 快照表。

理由：

- 当前数据结构以 `SQLite + uploads/` 为中心，直接在 `documents / folders` 上增加删除元数据，最贴近已有架构。
- 第一阶段已经有集合页视图骨架，回收站可以沿用同一套中栏切换模型。
- 恢复时不需要重建一套独立快照结构，能最大程度复用现有文档、文件夹、标签和附件关系。

## Product Shape

### 回收站

- 回收站是 `当前空间独立` 的中栏视图。
- 左侧原有 `回收站` 按钮保留位置，但点击后进入真实回收站视图。
- 回收站与 `所有笔记 / 收藏夹` 同级，属于一种中心视图，而不是左树的一部分。

### 删除语义

`文档删除`

- 从“立即删除”改成“移到回收站”。
- 删除后文档立即从左树、集合页、搜索结果和分享访问中消失。

`文件夹删除`

- 采用“文件夹包”语义。
- 删除文件夹时，根文件夹、其所有后代文件夹、以及其下所有文档进入同一个回收站包。
- 回收站里显示一个文件夹条目，而不是拆散成多个文档条目。

`空间删除`

- 继续保持当前硬删除。
- 本期不把空间纳入回收站。

### 回收站列表

回收站按 `删除时间倒序` 展示，只显示根条目：

- 文档根条目：标题、原目录、删除时间
- 文件夹根条目：名称、删除时间、包含的文件夹数和文档数

每条提供：

- `恢复`
- `彻底删除`

顶部提供：

- `清空回收站`

### 设置弹窗

- 点击左侧 `设置`，不再展开下拉设置面板。
- 应用级打开一个独立 `SettingsModal`。
- 当前第三期已经完成的数据工具全部迁移进弹窗：
  - 打开数据目录
  - 创建备份
  - 从备份恢复
  - 重建搜索索引
  - 清理孤儿附件

## Attachment Lifecycle

附件采用“两段式删除”：

- `移到回收站`：不删除附件
- `恢复`：不移动、不重写附件
- `彻底删除 / 清空回收站`：物理删除对应文档附件目录

保留第三期已完成的 `清理孤儿附件` 作为兜底维护工具。

这意味着：

- 文档恢复不需要做附件搬运
- 文件夹包恢复也不需要做附件搬运
- 真正的文件删除只发生在永久删除阶段

## Technical Design

### Data Model

在 `documents` 与 `folders` 表新增：

- `deleted_at TEXT`
- `trash_root_id TEXT`

语义：

- 活跃记录：`deleted_at IS NULL`
- 单独删除文档：`trash_root_id = document.id`
- 删除文件夹包：根文件夹、后代文件夹、后代文档全部写入同一个 `trash_root_id = 根文件夹 id`

这样回收站根条目可被稳定识别：

- 文档根条目：`deleted_at IS NOT NULL AND trash_root_id = id`
- 文件夹根条目：`deleted_at IS NOT NULL AND trash_root_id = id`

### Repository Contract

正常查询默认只返回未删除记录：

- 文档列表
- 文档详情
- 文件夹列表
- 工作区快照

新增回收站相关能力：

- 列当前空间回收站根条目
- 软删除文档
- 软删除文件夹包
- 恢复回收站条目
- 彻底删除回收站条目
- 清空当前空间回收站

### Frontend State

把 `activeCollectionView` 扩展为：

- `tree`
- `all-notes`
- `favorites`
- `trash`

设置弹窗状态不再挂在空间切换面板内部，而是提升到 `AppShell` 级别。

### Search And Shares

已进入回收站的文档：

- 不应出现在工作区搜索中
- 不应继续可通过分享链接访问

策略：

- 软删除时从搜索索引中移除对应文档
- 恢复时重新写回索引
- 永久删除时维持删除状态
- 分享接口通过 `documents.getById` 的“只返回未删除记录”约束自然失效

### Restore Rules

`恢复文档`

- 清空该文档的 `deleted_at` 与 `trash_root_id`
- 继续挂回原 `folder_id`

`恢复文件夹包`

- 还原根文件夹、所有后代文件夹、所有后代文档
- 全部清空 `deleted_at` 与 `trash_root_id`
- 继续使用删除前已有的 `parent_id / folder_id`

## Settings Modal Design

`SettingsModal` 采用轻量应用层弹窗：

- 打开时遮罩页面
- 支持关闭按钮与遮罩关闭
- 展示当前存储状态、最近写入、数据目录路径
- 展示数据工具动作和反馈

空间切换浮层只保留：

- 切换空间
- 新建空间
- 重命名当前空间
- 删除当前空间
- 回收站入口
- 设置入口

不再在浮层内部承载完整设置内容。

## Error Handling

- 恢复不存在的回收站条目：返回友好失败提示，不破坏当前状态
- 永久删除失败：反馈失败，保留现有回收站内容
- 清空回收站过程中单个附件目录删除失败：继续清理其他目录，并汇总失败数
- 如果目标原目录已被恢复前手动删除：
  - 文档仍按原 `folder_id` 恢复
  - 若父文件夹不存在，则恢复到根目录

## Testing Strategy

1. Repository / smoke
- 文档软删除后不会出现在正常列表里
- 文件夹整包软删除后不会出现在树与快照里
- 恢复后文档和文件夹回到原位置
- 永久删除会物理移除记录

2. Search / share
- 已软删除文档不会出现在搜索结果
- 已软删除文档的分享链接返回 not found
- 恢复后重新可见

3. React UI
- 左树删除按钮文案和行为改成“移到回收站”
- 点击 `回收站` 后中栏显示回收站页面
- 恢复、彻底删除、清空回收站动作可见并有反馈
- 点击 `设置` 后打开弹窗，而不是展开下拉设置面板

## Acceptance

1. 文档删除后能在当前空间回收站中找到并恢复。
2. 文件夹删除后以“文件夹包”形式出现在回收站中，并能整体恢复。
3. 已进回收站的文档和文件夹不会出现在左树、集合页、搜索和分享访问中。
4. 彻底删除会连带移除对应文档附件目录。
5. 点击 `设置` 会打开独立弹窗，第三期数据工具全部可用。
