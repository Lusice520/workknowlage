# 需求文档规则

WorkKnowlage 需求文档采用分层结构：PRD 定产品方向和范围，SPEC 管单功能细节，issue 管开发执行。不要把所有细节都塞进 PRD，也不要让 SPEC 脱离 PRD 范围单独漂移。

## 标准路径

- PRD 模板：`docs/requirements/PRD_TEMPLATE.md`
- PRD 成品案例 / 当前项目 PRD：`docs/requirements/PRD.md`
- SPEC 模板：`docs/requirements/specs/SPEC_TEMPLATE.md`
- SPEC 成品案例：`docs/requirements/specs/local_share_spec.md`
- 单功能 SPEC：`docs/requirements/specs/<feature_name>_spec.md`
- 开发 issue：`docs/requirements/issues/<NN>-<slug>.md`，或该功能约定的 issue 目录

## 标准文档集合

WorkKnowlage 的需求文档体系包含 5 类核心 Markdown：

1. PRD 模板：定义项目级 PRD 的标准结构。
2. SPEC 模板：定义单功能规格说明书的标准结构。
3. 需求文档规则：说明 PRD、SPEC、issue 如何分层。
4. PRD 成品案例：展示模板如何应用到 WorkKnowlage 项目。
5. SPEC 成品案例：展示模板如何应用到具体功能。

不要把成品案例当模板。创建新 PRD 或 SPEC 时，先从 `PRD_TEMPLATE.md` 或 `SPEC_TEMPLATE.md` 开始，再参考案例的质量和写法。

## PRD 职责

项目级 PRD 负责：

- 产品定位
- 目标用户和业务场景
- 产品目标和成功指标
- 现有业务流程、目标业务流程和需求管理流程
- 系统架构图、信息结构图和功能结构图
- 功能范围清单
- SPEC 管理总清单
- 段落式需求说明
- 模块范围
- 里程碑
- 产品级风险和待确认问题

项目级 PRD 不应该写：

- 每一个按钮
- 每一个字段
- 每一条提示文案
- 每一个表格列
- 每一个边界交互细节
- 冗长的文档操作手册；只在必要时保留简短需求流程，用来解释 PRD -> SPEC -> issue 分层

## SPEC 职责

单功能 SPEC 负责：

- 背景与目标
- 方案取舍
- 产品形态与范围边界
- 功能列表
- 流程正文说明和流程图
- 页面 / 状态说明
- 字段、列表列、查询条件和表单
- 交互说明
- 提示文案
- 特殊业务规则
- 异常和兜底处理
- 数据记录
- 验收标准
- 待确认问题

单功能产品设计应合并进 SPEC，避免默认另建长期 design doc。SPEC 应保留有价值的设计判断、被放弃方案和采用原因，但不默认写文件级实施拆解。

执行计划主要是开发执行期工具。只有开发复杂、跨会话、需要交接、需要并行拆分，或用户明确要求先评审实现路径时，才单独创建执行计划。

## 什么时候创建 SPEC

满足任一条件时，应创建单功能 `<feature_name>_spec.md`：

- 功能包含多个状态、按钮、字段、弹窗、浮层或页面。
- 功能涉及本地数据、编辑器行为、导出、分享、搜索或迁移。
- 功能包含用户可见错误状态或提示文案。
- 功能曾被用户反复反馈，或需求存在歧义。
- 功能需要继续拆成开发 issue。

## 功能清单和需求说明

PRD 中的功能清单是产品范围索引，回答“当前范围包含哪些模块和功能组”。它不替代需求说明。

需求说明应使用面向产品的段落，说明：

- 这个模块服务什么用户需求
- 大致需要提供哪些能力
- 当前阶段明确不做什么
- 主要产品风险或交付风险是什么

字段、按钮、提示文案、状态表和异常处理细节放入 SPEC。

## PRD 顺序和图示

WorkKnowlage PRD 应先讲流程，再讲系统清单。推荐顺序：

1. 背景和产品定位
2. 目标用户和业务场景
3. 业务流程与需求流程
4. 系统概述
5. 功能清单
6. SPEC 管理总清单
7. 段落式需求说明

系统概述至少包含：

- 系统架构图
- 信息结构图
- 功能结构图

核心流程应尽量同时包含正文说明和 Mermaid 流程图。不要用一张压缩总览图替代多个命名核心流程。

每个核心流程应包含：

- 一段正文，说明用户意图、关键动作和预期结果
- 一张 Mermaid 流程图，展示顺序、分支和异常路径

纯步骤列表可以作为辅助，但不应成为核心流程的唯一表达。

## SPEC 流程章节

SPEC 的第 3 章也应使用“正文说明 + Mermaid 流程图”。不要只写箭头式流程。

每个重要流程或分支应说明：

- 用户意图
- 触发点
- 系统处理
- 成功结果
- 失败或兜底行为
- Mermaid 流程图

## PRD 内的 SPEC 管理总清单

每份 WorkKnowlage 项目 PRD 都应在功能清单之后、需求说明之前包含 SPEC 管理总清单。

该表应让读者看到：

- 哪些模块或功能已有 SPEC
- 每个 spec 文件在哪里
- 当前 SPEC 状态是什么
- 优先级是什么
- 哪些计划中的 SPEC 还未创建

PRD 内的 SPEC 表是管理索引，不是功能细节本身。不要把字段表、提示文案、状态机或详细异常处理复制进 PRD 表格。
